Expand description

Write access to an N-dimensional lattice.

Extends Lattice<T> with mutation methods. Corresponds to the non-const portion of C++ Lattice<T> (putSlice, set, etc.).

Not all lattice implementations are writable — for instance, expression lattices are read-only. The Lattice::is_writable method indicates whether mutation is supported.

fn put_at(&mut self, value: T, position: &[usize]) -> Result<(), LatticeError>

Sets a single element at the given position.

Returns an error if position is out of bounds or the lattice is not writable.

Source

fn put_slice( &mut self, data: &ArrayD<T>, start: &[usize], ) -> Result<(), LatticeError>

Writes a rectangular sub-region into the lattice.

data is placed at the given start position. The data shape determines the extent of the region written. No stride is applied (contiguous write).

Source

fn set(&mut self, value: T) -> Result<(), LatticeError>

Sets all elements of the lattice to value.

Corresponds to C++ Lattice<T>::set(value).

Executes f while a mutable traversal-specific cache hint is active.

Paged backends can override this to temporarily retune internal tile caches for a known mutable access pattern, then restore the previous setting before returning.

Source

fn apply(&mut self, f: impl Fn(&T) -> T) -> Result<(), LatticeError>
where Self: Sized,

Applies a function to every element of the lattice in place.

This is a convenience method that reads the entire lattice, maps f over each element, and writes the result back. Subclasses may override for more efficient chunk-based application.

Source

fn copy_data(&mut self, source: &dyn Lattice<T>) -> Result<(), LatticeError>

Copies all data from source into this lattice.

The source lattice must have the same shape as this lattice.

fn for_each_chunk_mut( &mut self, spec: TraversalSpec, f: impl FnMut(&mut ArrayD<T>, &TraversalCursor) -> Result<(), LatticeError>, ) -> Result<(), LatticeError>
where Self: Sized,

Applies f to each chunk selected by spec, writing modified chunks back automatically after each successful callback.

This provides a simpler mutable traversal API than the explicit iterator/write-back pattern while still operating chunk by chunk.
