What This Diagram Should Answer

Primary question Does the current implementation split functionality across too many functions, modules, classes, scripts, or job wrappers?
Secondary question Once the structure is cleaner, where should optimization work look first: GPU kernels, worker threads, I/O overlap, product writes, or scheduling?

A plain dataflow diagram is not enough for this. The useful diagram is a dataflow plus ownership overlay. Every node names the transformation and the current implementation owner. Mode splits are explicit: MFS, cube, MTMFS, mosaic, W-projection, and AW-style paths. Resource boundaries are shown separately from source-code boundaries.

Package Overlays

casa-rs

The Rust structure has a useful hard boundary: casa-imaging is a pure core that consumes prepared batches and emits products, while casars-imager owns MeasurementSet I/O, mode routing, coordinates, masks, and product writing. Most near-term collapse pressure is therefore in the adapter/orchestration layer.

Review signals
  • Keep the pure casa-imaging prepared-batch boundary.
  • Question adapter-level mode dispatch, cube orchestration, and product-writing spread in casars-imager.
  • Standard/W-term gridding is intentionally grouped; mosaic PB projection is a separate implementation path.

CASA C++

CASA C++ is intentionally layered. The Python task and helper classes own task-level control, the synthesisimager tool exposes the C++ imager object, and SynthesisImager delegates actual image and visibility operations through mapper, image-store, normalizer, deconvolver, and FTMachine families.

Review signals
  • Use CASA C++ as a semantic oracle, not a Rust shape to copy directly.
  • The separate FTMachine subclasses are real mode/projection boundaries.
  • Understand Python-helper versus C++ ownership before mirroring normalization or PB-correction behavior.

LibRA

LibRA keeps much of the CASA synthesis vocabulary, but its distinctive split is at the workflow and resource boundary. The ImageSolver wrappers decompose imaging into job modes such as weight, PSF, residual, model, gather, normalize, restore, and dirty image.

Review signals
  • Most extra splitting is a resource/workflow split for local, Slurm, HTCondor, and GPU/no-GPU execution.
  • The algorithm classes name distributed image-solver steps directly: PSF, residual, model prediction, writeback, gather, normalize, and restore.
  • The resampler/worklet boundary is the strongest contrast for later worker-thread and GPU planning.

How To Read Split Pressure

This diagram style can show whether functionality is spread across too many functions or modules, but only if reviewers distinguish real boundaries from control-flow-only splits.

Keep a split when it is a real boundary
  • different data shape, such as raw MS rows versus prepared scalar visibility batches
  • different resource location, such as local process versus Slurm/HTCondor job
  • different external contract, such as Python task syntax versus Rust core API
  • different algorithm family, such as standard gridder versus AW-projection
  • different persistence contract, such as image arrays versus CASA image tables or MODEL_DATA
Question a split when it only changes control flow
  • several functions pass the same request and data shape through mode-specific branches
  • product writing branches know too much about how each mode was executed
  • unsupported-mode checks are duplicated at task, adapter, and core layers
  • cube and MFS paths run equivalent per-plane loops in different modules
  • diagnostics and timings are assembled in many places after the same stage

Immediate casa-rs hypothesis

Do not collapse the pure core. Keep casa-imaging as the prepared-batch implementation boundary and reduce pressure in casars-imager.

Likely refactors

Extract a prepared run plan for mode dispatch, plane looping, and clean-mask seeding. Extract product emission behind a product manifest or writer abstraction.

Before optimization

Make gridder/projection planning a named boundary before adding fast W/A/AW kernels, GPU implementations, or multi-worker execution.

Optimization Map

Once the shape is stable, this same diagram can be annotated with timings, memory pressure, and resource ownership.

Likely casa-rs measurement points
  • MS column loads in the bounded source stream readers
  • weighting density construction
  • standard gridder and W/screen projector sample loops
  • FFT and normalization
  • residual refresh/degrid during major cycles
  • CASA image-table product writes and preview generation
  • optional MODEL_DATA writes
Likely optimization experiments
  • streaming or chunked prepared batches to reduce peak memory
  • overlap MS column reads with weighting or gridding
  • Rayon or scoped-worker parallelism behind gridder/projector traits
  • GPU kernels behind a projection-plan boundary
  • separate product emission workers after core results are available
  • reuse LibRA-style resampler/worklet ideas for grid/degrid hot loops
