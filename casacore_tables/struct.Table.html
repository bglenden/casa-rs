A casacore table — the fundamental persistent data container.

A Table holds a set of named columns (scalar or array) plus keyword metadata, backed either by on-disk storage or an in-memory buffer.

§Lifecycle

  • Open — Table::open reads an existing table directory.
  • Create — [Table::create] builds a new table from a schema.
  • Save — Table::save flushes deferred writes to disk.

§C++ equivalent

casacore::Table — the main user-facing class in the Tables module.

pub fn open(options: TableOptions) -> Result<Self, TableError>

Opens an existing table from disk.

Reads schema, keywords, column keywords, and table metadata eagerly from the directory identified by options.path(). Row payloads are materialized lazily on first row access.

This keeps open() much closer to C++ casacore’s lazy table-open behavior for large persistent tables such as MeasurementSets.

If the on-disk table is a reference table (RefTable type marker), the parent table is opened automatically and the referenced rows are materialized into this table.

Source

pub fn open_metadata_only(options: TableOptions) -> Result<Self, TableError>

Opens only table metadata from disk, without materializing rows.

This loads schema, keywords, column keywords, table info, and data manager metadata, but intentionally leaves the row store empty.

It is primarily intended for layered APIs such as paged images that need cheap access to table metadata while reading pixel data through a separate tiled I/O path. Row- and cell-level accessors on the returned table therefore behave as if the table has zero rows.

This is a Rust-specific optimization; casacore C++ does not expose a direct equivalent on Table.

Source

pub fn save(&self, options: TableOptions) -> Result<(), TableError>

Saves the table to disk.

Validates the table against its schema (if any), then writes all rows, keywords, column keywords, and schema to the directory specified by options.path(). The data manager format is determined by options.data_manager(). The directory need not exist beforehand; the storage layer creates it.

Returns TableError::Storage on I/O failure.

Source

pub fn save_metadata_only( &self, options: TableOptions, ) -> Result<(), TableError>

Saves only table metadata back to disk without rewriting row storage.

This updates table.dat keyword records, column keyword records, and table.info, while preserving the existing on-disk data-manager layout. It is intended for layered APIs such as tiled images that mutate table metadata but keep their payload in a separate storage manager.

This is a Rust-specific optimization and currently supports only plain tables on disk.

Source

pub fn save_with_bindings( &self, options: TableOptions, bindings: &HashMap<String, ColumnBinding>, ) -> Result<(), TableError>

Save the table with per-column data manager bindings.

Columns listed in bindings are stored using their specified DM; all other stored columns use the default DM from options.

This allows mixing storage managers within one table, for example scalars in StandardStMan and arrays in TiledColumnStMan.

§Example
use std::collections::HashMap;
use casacore_tables::{Table, TableOptions, DataManagerKind, ColumnBinding};

let table = Table::default();
let mut bindings = HashMap::new();
bindings.insert("DATA".to_string(), ColumnBinding {
    data_manager: DataManagerKind::TiledColumnStMan,
    tile_shape: Some(vec![4, 32]),
});
table.save_with_bindings(
    TableOptions::new("/tmp/my_table"),
    &bindings,
)?;

pub fn open_with_lock( options: TableOptions, lock_opts: LockOptions, ) -> Result<Self, TableError>

Opens an existing table from disk with locking.

Behaves like open but also creates or opens the table.lock file and acquires a lock according to the given LockOptions.

  • LockMode::PermanentLocking: acquires a write lock immediately; fails if unavailable.
  • LockMode::PermanentLockingWait: acquires a write lock, waiting indefinitely.
  • LockMode::AutoLocking: acquires a read lock immediately; write operations temporarily acquire/release a write lock.
  • LockMode::UserLocking: no lock is acquired until lock() is called.
  • LockMode::NoLocking: equivalent to open().

C++ equivalent: Table(name, TableLock(...), Table::Old).

Source

pub fn lock( &mut self, lock_type: LockType, nattempts: u32, ) -> Result<bool, TableError>

Acquires a lock on the table.

Re-reads the table data from disk if another process modified it since the last lock was held.

nattempts: number of lock attempts. 0 means wait indefinitely, 1 means try once without waiting.

Returns true if the lock was acquired, false if it could not be acquired within the given attempts.

C++ equivalent: Table::lock(type, nattempts).

pub fn sort( &self, keys: &[(&str, SortOrder)], ) -> Result<RefTable<'_>, TableError>

Sorts the table by the given key columns, returning a RefTable with the rows in the new order.

The result is an indirect sort: no data is moved, only the row index permutation changes. The returned RefTable has row_order = false (not in original ascending order).

Only scalar columns with a total ordering can be sort keys. Complex columns are rejected. This matches C++ Table::sort.

§Errors
  • TableError::SortNoKeys if keys is empty
  • TableError::SortKeyNotScalar if a key column is non-scalar
  • TableError::SortKeyUnsortable if a key column is Complex
  • TableError::UnknownColumn if a key column is not in schema
§C++ equivalent

Table::sort(columnNames, sortOrders).

Returns an iterator that groups rows by equal values in the key columns.

The table is first sorted by the key columns, then consecutive rows with equal key values are collected into TableGroup values. Each group contains the shared key values and the parent-table row indices for that group.

Unlike sort, this borrows the table immutably because it yields owned data rather than a mutable view.

pub fn query_result(&self, taql: &str) -> Result<QueryResult<'_>, TableError>

Executes a TaQL SELECT query and returns a QueryResult.

For simple SELECTs (only column references, no computed expressions), returns QueryResult::View with a zero-copy RefTable. For SELECTs with computed columns, GROUP BY, or aggregate functions, returns QueryResult::Materialized with an owned in-memory Table containing the evaluated result rows.

§Errors

Returns TableError::Taql if the query is invalid or execution fails.

§Examples
match table.query_result("SELECT * WHERE flux > 2.0").unwrap() {
    QueryResult::View(view) => assert_eq!(view.row_count(), 2),
    QueryResult::Materialized(mat) => {
        // Computed columns produce a materialized table
        let _ = mat.row_count();
    }
}

C++ equivalent: tableCommand() with a SELECT query.
