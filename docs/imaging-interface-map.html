casa-rs Imaging Interface Map

A "now vs target" map of public and private imaging interfaces. The goal is to expose where similar operations are split across app, MS, and core imaging layers, then give a conservative path to simplify without losing CASA semantics or performance.

Snapshot date: 2026-07-02. Evidence paths are relative to the casa-rs repository root.

Executive Read

The application crate is doing more than UI, task parameters, and output orchestration. It contains private computation paths for MS selection, channel planning, density accumulation, spectral interpolation, cube slab scheduling, primary-beam products, masks, and product assembly. Those are the highest-value migration candidates.

The existing intended boundary is still good: keep casa-imaging as the prepared-batch imaging core, keep casa-ms as the MeasurementSet and spectral-selection owner, and make casars-imager an application/runtime shell. The work is not "collapse everything into one function." It is to move domain computation out of the app crate behind a smaller set of library-owned plans.

CASA Mode Frame

CASA imaging is naturally multi-axis: data selection and visibility buffering, spectral shape, gridder/projection family, weighting, major/minor-cycle control, deconvolver, execution backend, and product persistence all vary independently. RadioAstronomyOracle hits from CASA workshop material line up with this view: joint mosaic imaging in tclean uses Cotton-Schwab major/minor cycles with deconvolver choices such as Hogbom, Clark, and multiscale; wideband imaging material lists Standard gridding, W-projection, wideband A-projection, joint mosaics, parallelization, cube, MFS, MT-MFS, faceting, Stokes, and multi-field as separate axes.

In the current Rust code, AW-style task aliases are accepted at the edge but map to W-projection behavior or reject unsupported combinations. No current W-stacking implementation was found. Treat W-stacking as a future W-term execution strategy sibling to W-projection, not as a special case in product writing or CLI dispatch.

Current Architecture

Today the imaging path is a hybrid: casars-imager is both the task/runtime adapter and a substantial preparation/execution coordinator, while casa-imaging owns most pure image-domain and grid-domain computation once visibility data has been shaped into its expected batches. This is why the system works, but also why similar operations appear in multiple forms.

The practical current architecture is therefore not just "app calls library." It is "app parses and routes, app reads and shapes MS data, app chooses one of several mode-specific preparation paths, core imaging executes the hottest shared work, and app assembles persisted products."

LibRA Interface Ideas

LibRA is useful here because it deliberately tries to expose radio astronomy algorithms through a shallower stack. Its README states the goal as direct access to core RA image-reconstruction algorithms with a simple interface and small dependency graph, and separates scientific code, standalone applications, and workflow scripts. The lesson for casa-rs is not to copy LibRA's shell/parfile surfaces, but to preserve its more explicit stage vocabulary and backend boundaries in typed Rust interfaces.

Proposed Architecture

The proposed shape is a layered dataflow and stage graph. The application crate remains responsible for user-facing contracts: task parameters, aliases, defaults, validation messages, progress, summaries, and output policy. The library crates own the computations: visibility selection and bounded reads in casa-ms; mode planning, weighting, gridding, projection, FFT, major/minor-cycle control, deconvolution, and backend selection in casa-imaging; and a narrow product writer consumes typed products without knowing how they were computed.

This is the main simplification: preserve the independent CASA imaging axes, but stop encoding each full combination as a separate app path. Instead, build a plan once, execute stages through a small number of library-owned contracts, and keep CPU, multi-worker, and Metal differences behind execution strategies.

Oracle refinement: the canonical replacement must not flatten everything to one owned VisibilityBatch. The target public surface is a neutral, bounded, replay-capable visibility source. Replayable row blocks, routed runs, fixed-tile payloads, and Metal grouped-input cache payloads can remain as private backend IR because they preserve performance properties that the current public variants were created to protect.

Oracle Iteration Refinements

The Oracle review agreed that the layering direction is right, but pushed the plan to be stricter. The central correction is that the refactor must reduce public and app-private surface without erasing internal performance shapes.

Do not flatten to owned batches. The new public API must preserve bounded replay, fixed-tile scheduling, cube-compatible clean-plane execution, and Metal grouped-input staging. Those shapes can remain as private backend IR; they should not remain public API.
Oracle point Plan update
Deletion must be first-class. Add a deletion ledger with current symbol, target owner, replacement, removal phase, and parity/performance gate.
StagePlan is too broad if it becomes a workflow engine. Treat stages as a closed typed vocabulary. Keep executable graph mechanics private until reused by multiple modes.
VisibilityResampler risks becoming a Rust FTMachine. Keep it narrow: bounded grid/degrid work only. Projection caches, FFT, normalization, worker reduction, Metal transfer, and clean control stay in separate private strategies.
Product writing was too broad. Split computed products and metadata from persistence: ImageProductSet and ImageProductMetadata in casa-imaging, table encoding in a writer, and naming/replacement policy in the app.
AW aliases should not silently mean W-projection. Once ProjectionPlan exists, unsupported AW should validate as unsupported or experimental rather than masquerading as WProject.
Backend fallback must be explicit. Add ExecutionCapabilities with selected backend, rejection reasons, worker count, device details, and explicit fallback policy. Replace boolean probes such as standard_mfs_metal_device_available().

API Budget And Gates

Public API Budget

  • One canonical run entrypoint: run_imaging_plan.
  • Optional session entrypoint for stepped/plane-keyed execution.
  • One visibility source contract with bounded replay semantics.
  • Plan, capability, product, and metadata types needed by apps and tests.
  • No public standard-MFS backend payloads unless a real external consumer is identified.

No Silent Fallback

  • Backend capability checks return structured reasons.
  • Fallback from Metal or multi-worker to CPU is explicit policy.
  • Environment variables may aid diagnostics but do not own user-facing behavior.
  • Backend logs include selected backend, worker count, tile shape, cache budget, and transfer bytes.
Gate Required evidence
API shrink Public symbols added/removed/demoted, app-private helpers deleted, wrappers remaining, and explicit removal phase for every wrapper.
Bounded streaming Max rows/channels/correlations per block, max resident visibility bytes, max resident routed/planned samples, concurrently retained blocks, replay mode, and confirmation that no full selected-MS materialization occurs.
Internal-shape preservation Weighted sample count, routed row/run count, planned run-block count, tile occupancy, per-block allocations, Metal grouped cache entries/bytes/hit rate, worker gather/reduction bytes.
CASA parity Selected rows, DDIDs/SPWs/fields/channels, frequency conversion, flags, weights/sigma, density grid, sumwt, PSF, dirty image, residual/model/restored image, cube coordinates, Taylor products, PB/pbcor, beam metadata, coordinate metadata, and history.
Clean session Prepare-plane, skip threshold, one-major-cycle, finish, minor stop reason, model update, and residual update equivalence.
Product metadata Semantic metadata equality plus serialized CASA coordinate-system equality: direction, spectral, Stokes, units, beam/restoring beam, mask metadata, PB/pbcor metadata, Taylor-term names and axes.

Migration Sequence

  1. Freeze gates and the deletion ledger. Every new abstraction must name the old public or private surface it will delete or demote, plus the parity and performance gate that permits removal.
  2. Introduce the neutral visibility source contract: VisibilitySource, VisibilityBlockStream, VisibilityBlockView, replay capability, and stream bounds. Do not flatten specialized routes to owned batches.
  3. Collapse public standard-MFS entrypoints behind a canonical run/session boundary. The first vertical slice is standard MFS dirty/PSF/clean plus the prepared-clean-plane lifecycle, not full cube. Keep planned run-block, fixed-tile, and Metal grouping internals private.
  4. Move MS selection and channel-read helpers into casa-ms: row selection, source partitions, selected physical channels, column read granularity, and bounded stream factories.
  5. Move app spectral/sample shaping into casa-imaging using the route vocabulary: ImagingSourceBlockView, SpectralRoutePlan, SourceChannelRoute, and OutputPlaneContribution.
  6. Move weighting and density accumulation into casa-imaging under WeightingPlan and ExecutionPlan.
  7. Split products and persistence: ImageProductSet and ImageProductMetadata in casa-imaging, ImageTableWriter for serialization, app for naming and replacement policy.
  8. Unify clean control, including outliers and MT-MFS. Treat run_mtmfs as a transitional island with an explicit deletion phase until it becomes a spectral-basis/deconvolution configuration of the same session architecture.
  9. Expand projection families after the stream, spectral-route, product, and clean boundaries are stable. Add W-stacking and AW as projection strategies, not app-level runners or aliases that silently route through W-projection.
Performance guardrail: every phase must preserve bounded streaming, stage timings, backend-plan logging, internal standard-MFS route shapes, and product metadata parity. Avoid moving code in a way that reintroduces full visibility materialization or makes CPU/Metal timings less attributable.
